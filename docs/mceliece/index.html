Classic McEliece In Tyr

From the big picture down to the hot loops

Tyr's McEliece backend is a parameterized, mostly pure-Nim implementation of the Classic McEliece Niederreiter KEM, focused on the 6688128f, 6960119f, and 8192128f fast variants. The implementation starts from the usual code-based story: choose a hidden binary Goppa code, publish a systematic parity-check matrix, encrypt by producing a sparse error vector, and decrypt by reconstructing that error from the syndrome.

The interesting engineering work is not in the slogan. It is in the details: parameter-specific reduction polynomials, non-byte-aligned row packing, constant-time sorting and Benes control bits, AVX2-assisted key generation, and making the decapsulation path stay branchless while still being fast.

1. System View

At a high level, the code implements the Niederreiter flavor of Classic McEliece. Instead of encrypting a message directly, encapsulation samples a sparse error vector \(e\) of Hamming weight \(t\), computes a syndrome \(c\), and hashes a preimage that contains \(e\) and \(c\). Decapsulation reconstructs the same error vector from the secret Goppa structure and hashes the matching preimage.

Key Generation

Hide structure, publish a systematic matrix

Build an irreducible Goppa polynomial \(g(x)\), a secret support \(L\), and a permutation \(\pi\). From those, derive a large parity-check matrix, row-reduce it into systematic form, and serialize the public tail.

Encapsulation

Sample a sparse error and compute its syndrome

Choose a random bit vector \(e \in \{0,1\}^n\) with exactly \(t\) ones, then compute \(c = H e^T\) using the packed public key. The shared secret is \(K = \text{SHAKE256}(1 \parallel e \parallel c)\).

Decapsulation

Use the hidden code to recover the same error

Rebuild the support via the Benes network, compute the syndrome sequence, recover the locator polynomial with Berlekamp-Massey, evaluate roots, and reconstruct the packed error vector. A masked select then chooses the correct preimage without branching on the validity bit.

Public side

Tyr stores the public key as the packed systematic tail of the parity-check matrix:

\[ H_{\text{sys}} = \left[I \mid T\right] \]

Only the \(T\) part is serialized. That is why the row-packing code in pk_gen.nim matters so much.

Secret side

The secret key layout in this implementation is:

secretKey = seed[32] || pivots[8] || irr || controlBits || s

Here irr is the irreducible Goppa polynomial, controlBits drive the Benes network, and s is the fallback secret portion used by the decapsulation preimage logic.

Important implementation fact: Tyr's current SIMD work is concentrated in pk_gen.nim. The AVX2 path helps key generation, especially matrix fill and row xor. It is not a broad "everything is vectorized" port of the full upstream vec/avx implementation.

2. Small Worked Examples

Real Classic McEliece uses \(n\) in the thousands and arithmetic in \(GF(2^{13})\). That is not something people do by hand. The point of the examples below is to keep the algebra recognizable, not to replicate full production sizes.

Example A: a toy syndrome over \(GF(2)\)

Take the tiny parity-check matrix

\[ H = \begin{bmatrix} 1 & 0 & 1 & 1 \\ 0 & 1 & 1 & 1 \end{bmatrix} \]

and the error vector

\[ e = \begin{bmatrix} 0 \\ 1 \\ 0 \\ 1 \end{bmatrix}. \]

Then

\[ c = H e^T = \begin{bmatrix} 1 \cdot 0 + 0 \cdot 1 + 1 \cdot 0 + 1 \cdot 1 \\ 0 \cdot 0 + 1 \cdot 1 + 1 \cdot 0 + 1 \cdot 1 \end{bmatrix} \bmod 2 = \begin{bmatrix} 1 \\ 0 \end{bmatrix}. \]

Tyr's encrypt.nim does exactly this kind of operation, just on the packed systematic matrix tail and on error vectors of weight \(t\).

Example B: finite-field multiplication by hand

To see how polynomial arithmetic works, use a smaller field: \(GF(2^4) = GF(2)[x]/(x^4 + x + 1)\). Multiply \[ a(x) = x^3 + x + 1,\qquad b(x) = x^2 + 1. \]

First expand:

\[ a(x)b(x) = (x^3 + x + 1)(x^2 + 1) = x^5 + x^3 + x + 1 \] because the two \(x^2\) terms cancel in characteristic two.

Now reduce \(x^5\) modulo \(x^4 + x + 1\): \[ x^4 \equiv x + 1 \quad \Rightarrow \quad x^5 \equiv x(x+1)=x^2+x. \] So the final result is \[ x^3 + x^2 + 1. \]

Tyr's gf.nim does the same style of multiply-and-reduce operation, but in \(GF(2^{13})\). One subtle point was that 6960119f does not use the same reduction pattern as 6688128f and 8192128f. That variant-specific difference had to be parameterized correctly.

Example C: what a control-bit layer does

The verification helper in controlbits.nim repeatedly applies layers of conditional swaps. For a tiny stride-1 layer on 8 positions, suppose the input is

\[ p = [0,1,2,3,4,5,6,7] \]

and the layer control bits are \[ [1,0,1,0]. \]

That means:

  • swap positions \((0,1)\)
  • keep \((2,3)\) unchanged
  • swap \((4,5)\)
  • keep \((6,7)\) unchanged

so the output becomes

\[ [1,0,2,3,5,4,6,7]. \]

Real Benes networks use many such stages, recursively arranged. The control-bit problem is to synthesize all those layer decisions so that the whole network realizes the target permutation \(\pi\).

Decode-side syndrome sequence: the explicit formula most closely mirrored by synd.nim is \[ S_j = \sum_{i=0}^{n-1} e_i \cdot \frac{L_i^j}{g(L_i)^2}. \] The code computes this by masking each contribution with the bit \(e_i \in \{0,1\}\), then repeatedly multiplying by \(L_i\).

3. Variant Table

The parameter table lives in params.nim. Tyr currently targets only the f variants, where the f means fast. These trade the regular variants' structure for faster computation, especially in the support and polynomial arithmetic paths.

Variant \(m\) \(n\) \(t\) Rows \(t \cdot m\) Systematic tail columns \(n - tm\) Reduction terms
6688128f 13 6688 128 1664 5024 \([7,2,1,0]\)
6960119f 13 6960 119 1547 5413 \([8,0]\)
8192128f 13 8192 128 1664 6528 \([7,2,1,0]\)

The second row is the awkward one. Since \(1547 \bmod 8 \neq 0\), the public-key packing logic for 6960119f has a non-byte-aligned tail. That detail drove one of the most important compatibility fixes against upstream behavior.

5. Module Details

5.1 params.nim: the implementation starts with the parameter table

The cleanest improvement here was also one of the most important correctness fixes: the code stopped pretending that all three variants share the same reduction polynomial. That is false. In particular, 6960119f must reduce with [8,0], not with the [7,2,1,0] pattern used by the other two.

This module also exposes all derived lengths used elsewhere: row bytes, syndrome bytes, irreducible polynomial bytes, control-bit bytes, and the dimensions of the systematic public-key tail. Without that single source of truth, the row-packing code becomes fragile very quickly.

5.2 util.nim: bit operations, masks, and safe wiping

This file is intentionally small. It supplies little-endian load/store helpers, bit reversal, constant-time zero/nonzero masks, and now the clearSensitiveWords helper, which uses volatile stores for POD buffers.

The zeroization work is not cosmetic. Key generation and decapsulation allocate large scratch buffers that contain support elements, locator polynomials, packed matrices, permutation material, and transient seeds. Those are now wiped on scope exit in the modules that allocate them.

5.3 gf.nim and sk_gen.nim: field arithmetic and irreducibility

The field layer implements branch-free arithmetic in \(GF(2^{13})\). The point is not merely to multiply elements; it is to do so without table lookups indexed by secrets and without variant mixups. The polynomial multiply helper GFmul is parameterized by the selected variant, so the same code can handle both \[ x^t + x^7 + x^2 + x + 1 \] and \[ x^t + x^8 + 1 \] style reductions.

sk_gen.nim then builds the minimal polynomial of the sampled field elements with Gaussian elimination. If the matrix is singular, key generation retries with fresh seed material. That retry is a keygen-only event, not a decapsulation oracle surface.

5.4 root.nim, synd.nim, and bm.nim: the decode math core

These three files together are the mathematical center of decapsulation:

  • root.nim evaluates polynomials over the support.
  • synd.nim builds the \(2t\)-term syndrome sequence.
  • bm.nim runs Berlekamp-Massey to recover the error-locator polynomial.

The important engineering constraint here is constant-time behavior. Berlekamp-Massey in particular is written with masked updates rather than "if discrepancy then update" style branching.

5.5 sort.nim: the unexpectedly important part

The file looks modest, but it ended up being one of the main performance levers. Tyr has constant-time bitonic-style sorters for both uint64 and int32. The uint64 sort is used in public-key generation to order permutation words; the int32 sort is used repeatedly inside control-bit synthesis.

A surprising compiler insight came out of this file: the monolithic Nim sorter compiled into much worse code than a chopped version using small inline helpers such as Top, LeadPass, Cascade, and TailPass. That restructuring brought the Nim uint64 sorter close to the C helper, and doing the same to int32SortRaw largely rescued pure-Nim controlbits performance.

5.6 controlbits.nim: the Benes synthesis bottleneck

This file synthesizes the control bits for the Benes network from the target permutation. It still keeps two relevant paths:

  • a local C helper path mirroring the upstream clean algorithm
  • a pure-Nim unchecked path used for profiling and comparison

Historically this was the biggest keygen hotspot. The first instinct was to split the recursion into smaller helpers. That made the code easier to instrument but barely moved runtime. Profiling with Otter then showed the real issue: cbRecursion was hot inclusively, but the heaviest leaf routine was int32SortRaw. Optimizing the sorter mattered much more than reshaping the recursion itself.

Another useful negative result came from this file: splitting cbCompactLarge did not produce a meaningful gain. That is worth recording because it prevents repeated work later.

5.7 benes.nim and support.nim: rebuild the hidden support

The Benes layer is what turns abstract control bits into an actual support permutation. supportGen starts from bit-reversed field indices, lays them out bit-slice by bit-slice, and runs each slice through the Benes network. The output support \(L\) is then reconstructed field element by field element.

This code is heavy on fixed-structure data movement and transposes. It is performance sensitive, but it is also the kind of code where constant-time behavior is easier to keep because the network structure is fixed once the control bits are chosen.

5.8 pk_gen.nim: where most large speedups came from

Public-key generation is the broadest hotspot because it does many expensive things at once: evaluates the Goppa polynomial on the support, batch-inverts all those values, builds the packed matrix, applies masked Gaussian elimination, moves columns near the end, and finally packs the systematic tail into serialized public-key bytes.

The important changes here were:

  • using word-wise row xor rather than byte-by-byte xor in the elimination path
  • adding an AVX2 row-xor hot path with loadu/storeu
  • packing 64 values at a time with transpose64x64 in the AVX2 matrix fill path
  • fixing non-byte-aligned row packing for 6960119f
  • narrowing unaligned 64-bit loads into a safe copyMem fallback instead of broad typed aliasing

That last point matters for more than speed. Unaligned typed uint64* dereferences are tolerated by x86 hardware but are still UB-shaped enough to be an optimizer hazard. The current version keeps the aligned fast path and uses the fallback only where the buffer really is unaligned.

5.9 encrypt.nim: sparse error generation and packed syndrome evaluation

Encapsulation begins by generating a random weight-\(t\) error vector. The error indices are sampled, deduplicated, and packed into bytes. Then the code computes the public syndrome directly from the packed public-key rows.

One useful optimization here was to avoid rebuilding a full temporary row buffer for every public-key row during syndrome computation. Instead, parity is accumulated directly from the stored packed rows. That helped encapsulation, although key generation still dominates the total runtime much more than encapsulation does.

5.10 decrypt.nim and operations.nim: constant-time decapsulation

Decapsulation is where timing discipline matters most. Earlier versions had clearly unsafe control flow:

  • early returns when the recovered error weight differed from \(t\)
  • breaking on the first syndrome mismatch
  • branching on the decapsulation validity bit when building the preimage

Those patterns were removed. The current code accumulates a branchless okMask and uses that mask to build the preimage for \[ K = \text{SHAKE256}(b \parallel e' \parallel c) \] without a secret-dependent branch. That change matters much more than a small speed delta.

6. Optimization Story

The optimization work on Tyr's McEliece implementation was deliberately incremental: isolate a hotspot, change one thing, measure, keep or revert. That matters because this code has several places where "obvious" changes either did nothing or regressed.

Area Problem Change Outcome
params + gf 6960119f needed different reduction behavior Parameterize reduction terms instead of sharing one hardcoded pattern Correctness fix and parity with upstream behavior
pk_gen Byte-wise elimination was too slow Use word-wise xor, plus AVX2 row xor where available Large keygen speedup
pk_gen Matrix fill was scalar and cache-heavy Pack 64 values at a time using transpose64x64 Good AVX2-side gain
pk_gen 6960119f row tail is not byte aligned Port the upstream row-packing logic for the non-aligned case Correct public-key serialization
sort Monolithic Nim sorter generated poor code Split into small inline passes Very large microbench improvement
controlbits Recursion looked hot but was not the true leaf bottleneck Profile with Otter, then optimize int32SortRaw Pure-Nim controlbits moved near the C helper
controlbits cbCompactLarge looked promising Split and benchmark No meaningful gain; reverted
encrypt Unnecessary temporary row rebuilds Compute syndrome parity directly from packed rows Encapsulation became leaner, but not the main global win
controlbits / sort Local C helpers existed, Nim ports lagged Port and compare one helper at a time Useful measurement discipline and better Nim implementations

Funny but true

One of the most useful findings was that controlbits itself was not the whole problem. The real issue was the still-unoptimized int32 sorter inside it. Once that sorter got the same chopped inline treatment as the uint64 version, the pure-Nim controlbits path became dramatically more competitive.

What did not pan out

Splitting the controlbits recursion for codegen reasons helped readability and made instrumentation easier, but it did not deliver a comparable runtime win. That is a good reminder that inclusive hotspot percentages are not the same thing as true leaf causes.

7. Security And Hardening Notes

7.1 Secret-dependent branching that had to go

The most important cleanup was on the decapsulation path. Returning early on wrong weight, breaking on the first mismatch, or branching on the final validity bit would all create unnecessary timing structure. The current implementation instead:

  • accumulates a branchless validity mask during decode
  • uses masked preimage selection in operations.nim
  • keeps decapsulation behavior uniform enough to avoid a trivial timing oracle

7.2 SIMD attack surface review

The AVX2 path was audited mainly in pk_gen.nim. Two conclusions matter:

  • no clear secret/message-dependent branching was found in the AVX2 intrinsics path itself
  • the SIMD work currently lives in key generation, not in attacker-driven decapsulation

That does not make it "done forever", but it does reduce the exposure compared with a vectorized decap path. The more immediate issue found there was the unaligned typed uint64 access pattern, which was tightened into an aligned fast path plus a narrow copy fallback.

7.3 Zeroization

Wiping transient secret buffers was added across the implementation, especially in:

  • pk_gen.nim for matrix and inversion scratch state
  • operations.nim for seed material, permutation state, and preimages
  • decrypt.nim for support, locator, syndrome, and comparison buffers
  • encrypt.nim for temporary error-generation buffers
  • benes.nim, bm.nim, and sk_gen.nim for internal scratch arrays

The goal was to zero memory early enough to matter while keeping the hot paths reasonable. Tyr now does this with clearSensitiveWords, which uses volatile stores on POD buffers. That is a practical compromise: much better than leaving large scratch arrays live, without rewriting every helper into a custom allocator.

Constant-time caveat: measuring wall-clock time or instruction counts can reveal leaks, but it cannot prove constant-time behavior on its own. The work here removed clear source-level secret branches and tightened memory handling. That is necessary, not the same thing as a full leakage proof.

8. What Still Resists Optimization

Still hard

  • fully matching the upstream vec/avx design rather than only parts of it
  • pushing the Benes/control-bit machinery much further without giving up safety or readability
  • making the pure-Nim controlbits path the obvious default in every build shape
  • getting more speed without reintroducing secret-dependent control flow

Current practical state

  • scalar Tyr is around parity with the local official AVX2-enabled liboqs build
  • AVX2 Tyr has recently stayed comfortably better than 1.0x against that same comparison
  • the biggest remaining wins are more likely to come from deeper vector work than from tiny scalar cleanup
  • security hardening did not destroy AVX2 competitiveness, which is the right tradeoff
The McEliece code in Tyr is a good example of why performance work should stay empirical. The biggest wins came from places that looked mundane at first glance: row packing, row xor, and constant-time sort codegen.

Closing Summary

If you remember only one picture, remember this one: McEliece in Tyr is mostly a story about moving structured binary data around correctly and quickly, while keeping the secret-dependent parts branchless.

The high-level cryptography is classical code-based KEM design. The implementation story is where the engineering lives: parameter tables, field arithmetic, matrix packing, Benes control bits, AVX2-assisted keygen, targeted zeroization, and careful decapsulation masking.
