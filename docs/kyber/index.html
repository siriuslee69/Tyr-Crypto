High-Level View

Kyber in Tyr-Crypto

The Tyr Kyber implementation is a pure-Nim ML-KEM backend. The optimization work followed a strict order: first line the control flow up with liboqs / PQClean, then remove Nim-specific overhead such as sequence churn and pass-by-value scratch, then add SIMD only where the end-to-end benchmark actually improved.

The main design rule was simple: do not trade away auditability for a tiny local speedup. A good optimization here reduces copies, shortens fixed hot loops, or moves dispatch out of inner loops. A bad optimization hides the data flow behind broad casts, secret-indexed tables, or complicated control flow.

Reference-first tuning The Nim path was brought closer to liboqs before deeper micro-optimization.
Fixed-size state Hot buffers were kept on fixed arrays and caller-owned scratch where possible.
Selective hardening Constant-time compare and selective secure wiping stayed part of the design.

System View

At the top level, Kyber is three pipelines sharing one polynomial engine:

  1. Key generation samples secret vectors, expands the public matrix, multiplies in the NTT domain, and packs keys.
  2. Encapsulation derives message and coins, samples ephemeral vectors, performs matrix-vector arithmetic, and packs ciphertext plus shared secret.
  3. Decapsulation unpacks, decrypts the IND-CPA layer, re-encrypts, compares branchlessly, and selects the final secret.
Mental model: Kyber performance is mostly the cost of moving between three representations: packed bytes, coefficient-domain polynomials, and NTT-domain polynomials. The winning optimizations reduced the number of times those representations had to be rebuilt.

That is why the final implementation work was not just "make the NTT faster". It also involved closer pack/unpack placement, caller-owned SHAKE output buffers, fixed-size scratch, and more direct alignment with the liboqs dataflow.

Module Map

Module Main Role Why It Matters
params.nim, types.nim Variant selection, constants, fixed-size records. Known sizes allow the rest of the implementation to stay on arrays instead of heap objects.
reduce.nim Montgomery and Barrett reduction. Every multiplication and many normalization steps pass through this code.
ntt.nim Forward and inverse NTT butterflies. Small stage-order improvements here translate into large end-to-end gains.
poly.nim Single-polynomial arithmetic, message conversion, and noise helpers. Contains many small fixed loops that become expensive because they are repeated everywhere.
polyvec.nim Vector arithmetic and matrix-vector multiply-accumulate. This is the real arithmetic hot zone for keygen and encapsulation.
cbd.nim, symmetric.nim Noise derivation and XOF plumbing. Poor buffer handling here silently taxes every high-level operation.
indcpa.nim IND-CPA keygen, encrypt, decrypt orchestration. This is where arithmetic, packing, and XOF use all meet.
operations.nim ML-KEM public API and decapsulation selection logic. Security-sensitive compare and final shared-secret selection happen here.
verify.nim, util.nim Constant-time compare / move and clearing helpers. These modules decide whether the optimized arithmetic remains safe to expose.

Hot Path by Operation

Key Generation

  1. Expand the public matrix \(A\) from the seed \(\rho\).
  2. Sample secret vectors \(s\) and error vectors \(e\).
  3. Transform operands into the NTT domain.
  4. Compute \(t = A \cdot s + e\).
  5. Reduce, normalize, and pack the public and secret keys.

The expensive parts here are matrix generation, NTT conversion, and vector multiply-accumulate. The final Tyr tuning pass improved all three mainly by reducing temporary copies and by making the accumulation path look much more like the liboqs reference structure.

Encapsulation

  1. Hash input material into message and coins.
  2. Expand the public matrix and sample the ephemeral vector \(r\).
  3. Transform into the NTT domain.
  4. Compute \(u = A^T r + e_1\) and \(v = t^T r + e_2 + m\).
  5. Compress and pack the ciphertext, then derive the shared secret.

Encapsulation cares both about arithmetic cost and about packing cost. Several wins came from keeping buffers caller-owned and from not bouncing between packed and unpacked form more often than the algorithm requires.

Decapsulation

  1. Unpack ciphertext and secret key material.
  2. Run IND-CPA decryption to recover the embedded message.
  3. Re-encrypt and compare in constant time.
  4. Use a branchless conditional move to select the final shared secret.

This is where speed and hardening meet directly. The compare and final secret selection remain branch-free, and the public "try decaps and return success" style surface was hidden again so the API cannot become a decapsulation oracle by accident.

Performance Problems and Fixes

1. The first useful optimization was structural alignment

The earliest profitable pass was not a clever arithmetic trick. It was moving the Nim implementation closer to the liboqs / PQClean flow:

  • pack and unpack close to the actual use sites,
  • caller-owned scratch for SHAKE output and temporary arithmetic state,
  • matrix-vector accumulation laid out in the same broad order as the reference C.

Once the two implementations solve the same subproblem in roughly the same order, the remaining gap becomes understandable. Without that step, "optimization" often becomes random motion.

2. Nim overhead had to be removed before SIMD mattered

  • Fixed arrays replaced avoidable sequence traffic.
  • Pass-by-value scratch was replaced with borrowed mutable buffers.
  • SHAKE helpers were rewritten around caller-owned output buffers.
  • Zeroing was split into regular clear and secure clear so only secret-bearing exits pay the full cost.
Design rule: a fixed-size buffer with explicit ownership usually beats a cast-heavy shortcut. It is faster enough, safer to audit, and does not hide layout assumptions.

3. NTT stage work was cleaned before being interleaved

Forward and inverse NTT stages gained speed from interleaving and from earlier normalization. The point was not to invent a new transform. The point was to shorten loop scaffolding and keep coefficient state in a form that reduces follow-up work.

  • Forward NTT stages were partially interleaved.
  • Inverse NTT normalization was moved earlier where it removed redundant follow-up work.
  • A redundant post-basemul reduction path was dropped once the new stage order made it unnecessary.

4. Cached basemul was kept only in a linear-access form

"Cached basemul" sounds dangerous if it means secret-indexed tables. That is not what was kept. The accepted version stores intermediate arithmetic state in fixed scratch slots while visiting coefficients in a fixed order. No secret decides which table row or cache line gets touched.

  • Good cache: fixed-order arithmetic scratch that mirrors the scalar computation.
  • Bad cache: secret-dependent lookup into one of many precomputed result tables.

5. SIMD only stayed where the benchmark justified it

The successful SIMD pieces were the ones that replaced real arithmetic hot loops:

  • vectorized cached basemul in polyvec.nim,
  • selected AVX2 arithmetic inside the NTT and reduction path,
  • shared SIMD Keccak helpers only where the batching model helped the full operation.

The broad x4 SHAKE experiment was rolled back because it looked attractive in isolation but made end-to-end Kyber slower. That rollback discipline mattered just as much as the kept wins.

6. Security hardening stayed selective

Several hardening steps were kept because they solved a real problem without destroying the benchmark picture:

  • branchless compare and conditional move remain the decapsulation core,
  • the public decapsulation-validity oracle was hidden again,
  • the OpenSSL SHAKE shortcut was gated behind a compile-time testing flag,
  • secure zeroing was added only at strategic exits, not inside arithmetic loops.

7. Summary of the tuning logic

Problem Accepted Solution Why It Stayed
Reference mismatch Move the Nim orchestration closer to liboqs first. It made later profiling and comparison much more meaningful.
Sequence churn Replace with fixed arrays and caller-owned scratch. Less overhead and much better ownership clarity.
Loop-heavy NTT scaffolding Interleave stages and normalize earlier. Lower loop overhead without changing the math.
Suspicious basemul cache shape Keep only fixed-order arithmetic scratch. Preserves speed while avoiding secret-driven lookups.
Tempting but weak SIMD ideas Rollback any SIMD path that loses end-to-end. The final target is whole-algorithm speed, not isolated microbenchmarks.

Security and Side-Channel Decisions

The security pass focused on side channels that are easy to create during optimization:

  • secret-dependent branching,
  • secret-dependent table lookup or cache selection,
  • public API surfaces that leak decapsulation validity,
  • leaving secret-bearing temporary state live longer than necessary.

The compare and final secret selection remain branch-free. The accepted cached basemul path uses linear access only. Secret-bearing buffers are wiped at strategic exits with secure clears, while public or short-lived scratch uses regular clears so the implementation does not turn into a wipe loop.

Important distinction: the word "cache" by itself does not tell you whether a construction is dangerous. The dangerous version is "pick one prebuilt table from many based on secret data". The kept version here is just fixed-order arithmetic scratch.

A second security design choice was to keep the fast OpenSSL-backed SHAKE256 path disabled by default and behind an explicit compile-time testing flag. That preserves the benchmark tool when needed, while keeping the default custom-crypto path self-contained.

Worked Math Examples

Example 1: One NTT butterfly

Take modulus \(q = 3329\), inputs \(a = 100\), \(b = 200\), and twiddle \(\zeta = 17\). A simple butterfly step computes

\[ t = \zeta \cdot b = 17 \cdot 200 = 3400 \equiv 71 \pmod{3329} \]

Then the outputs are

\[ a' = a + t = 171,\qquad b' = a - t = 29 \]

This is the basic shape repeated across the transform: multiply one side by a twiddle, then add and subtract in the ring.

Example 2: Rejection sampling from three bytes

Kyber often packs two 12-bit candidates out of three bytes. Take

\[ b_0 = \texttt{0x34},\qquad b_1 = \texttt{0x12},\qquad b_2 = \texttt{0xAB} \]

Then

\[ d_1 = b_0 + 256 \cdot (b_1 \bmod 16) = 52 + 256 \cdot 2 = 564 \]

and

\[ d_2 = \left\lfloor \frac{b_1}{16} \right\rfloor + 16 \cdot b_2 = 1 + 16 \cdot 171 = 2737 \]

Because both are below \(q = 3329\), both are accepted as coefficients.

Example 3: Why fixed-order cached basemul is safe

Suppose two coefficient pairs are represented as

\[ u = (3, 5),\qquad v = (7, 11) \]

A tiny fixed-order accumulation computes

\[ s = 3 \cdot 7 + 5 \cdot 11 = 21 + 55 = 76 \]

The important implementation point is not the value \(76\). It is that the scratch holding the intermediate products is visited in the same order every time. The code never selects "table 3 instead of table 5" based on a secret.
