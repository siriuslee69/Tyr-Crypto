High-Level View

Frodo in Tyr-Crypto

Tyr's Frodo backend is a pure-Nim implementation of the Round 3 FrodoKEM-976-AES KEM. The guiding idea is the original Frodo design: stay with generic LWE instead of ring structure, accept a larger matrix, and make the implementation explicit and auditable.

That design choice shapes everything in the code. Frodo does not spend most of its time in an NTT. It spends most of its time generating slices of a large public matrix \(A \in \mathbb{Z}_q^{n \times n}\) from seed_A, multiplying those slices by skinny secret matrices, and packing the results. The most important optimization work therefore aligned Tyr's streaming matrix path with the liboqs / OpenSSL structure rather than inventing a different algorithm.

Generic LWE The hot path is explicit matrix generation plus matrix multiply, not ring arithmetic.
One implemented variant The current code tracks frodo976aes only, because that is the tested and benchmarked target.
Selective hardening Constant-time compare and masked decapsulation selection stayed in the design; public-only matrix generation was allowed to use faster AES plumbing.

System View

At the top level, FrodoKEM-976-AES in this repo is three fixed pipelines:

  1. Key generation: derive seed_A, sample \(S\) and \(E\), compute \(B = A S + E\), then pack the public and secret keys.
  2. Encapsulation: sample \(\mu\), derive ephemeral noise, compute \(B' = S' A + E'\) and \(V = S' B + E''\), then form \(C = V + \text{encode}(\mu)\).
  3. Decapsulation: recompute \(W = C - B' S\), decode \(\mu'\), rebuild the expected ciphertext branchlessly, and select the final shared-secret preimage with a mask instead of a branch.
Mental model: Frodo performance is mostly the cost of moving through four representations in a fixed order: seed material, matrix slices, word matrices, and packed byte strings. The winning work reduced matrix-generation overhead and kept those transformations close to the exact call sites that need them.

For this variant, the fixed parameters are \[ n = 976, \quad \bar{n} = 8, \quad q = 2^{16}, \quad B = 3. \] That means \(A\) is large, but all secret and ciphertext-side matrices are narrow. Tyr takes advantage of that shape by streaming four public rows at a time for \(A S + E\) and eight public columns at a time for \(S' A + E'\).

Public objects

  • seed_A and the derived public matrix \(A\)
  • the packed public key \(pk = (seed_A, \text{pack}(B))\)
  • the ciphertext \(ct = (\text{pack}(B'), \text{pack}(C))\)

Because \(A\) is public-derived, the matrix-generation path is allowed to use the faster public-data AES helpers.

Secret objects

  • the keypair seed prefix \(s\) stored at the front of sk
  • the secret matrix \(S\)
  • sampled error matrices and decoded \(\mu'\)
  • the final preimage selected during decapsulation

These are the values that drive constant-time compare/select logic and the explicit clear loops at the end of keygen, encaps, and decaps.

Hot Path by Operation

Key Generation

  1. Split 64 bytes of explicit or generated randomness into the fallback secret prefix, seed material for seed_A, and seed material for \(S\) and \(E\).
  2. Expand seed_A with SHAKE.
  3. Expand pseudo-random words for \(S\) and \(E\), then map them through the CDF sampler.
  4. Compute \(B = A S + E \pmod q\) with the streaming mulAddAsPlusEStream path.
  5. Serialize pk = seed_A || pack(B) and sk = s || pk || S || H(pk).

The expensive part is step 4. Tyr does not materialize the full \(976 \times 976\) matrix in the production path. It generates four rows at a time from seed_A, multiplies them against the eight secret columns, and discards the row block once the contribution has been accumulated.

Encapsulation

  1. Hash \(H(pk)\) together with \(\mu\) to derive ephemeral seed material and a candidate shared-secret suffix.
  2. Expand and sample \(S'\), \(E'\), and \(E''\).
  3. Compute \(B' = S' A + E'\) with the streaming mulAddSaPlusEStream path.
  4. Unpack \(B\) from the public key, then compute \(V = S' B + E''\).
  5. Encode \(\mu\), add it to \(V\), pack the ciphertext, and hash \((ct || k')\) into the final shared secret.

Encapsulation has two different multiplication shapes: \[ B' = S' A + E', \qquad V = S' B + E''. \] The first one is expensive because it still has to generate slices of \(A\); the second is cheaper because \(B\) is only \(976 \times 8\), not \(976 \times 976\).

Decapsulation

  1. Unpack \(B'\) and \(C\) from the ciphertext and recover \(S\) from the secret key.
  2. Compute \(W = C - B' S\), then decode \(\mu'\).
  3. Recreate the same ephemeral seed derivation the sender should have used.
  4. Recompute \(B''\) and \(C''\) from \(\mu'\) and compare both to the received ciphertext.
  5. Use a constant-time selector to choose either the recomputed key material or the secret fallback prefix from sk, then hash \((ct || selected)\).

That last step is the Fujisaki-Okamoto style hardening point. Tyr does not return "valid / invalid" and then a secret. It always returns a shared secret and keeps the validity bit inside masked selection logic.

Implementation Decisions and Tradeoffs

1. Only frodo976aes is implemented right now

frodo/params.nim defines only one variant: frodo976aes. That was deliberate. This repo's local tests, KAT comparison, and benchmark work all centered on that variant, so the first goal was to make one path correct, benchmarkable, and competitive before multiplying the maintenance surface with the other AES and SHAKE families.

2. The production path streams \(A\) instead of materializing it

A full \(976 \times 976\) matrix of 16-bit words is about 1.9 MB. That is not impossible to allocate, but it is wasteful for the hot path because each operation only needs a small slice at a time. Tyr keeps the fully materialized generateMatrixA helper mainly as a straightforward reference shape, while keygen and encaps use the streaming row and column kernels.

Why this matters: the algorithm is not just "matrix multiply". It is "generate matrix slice, consume matrix slice, move on". Keeping the same streaming structure as the reference implementation makes both performance tuning and correctness auditing easier.

3. The matrix generator was aligned with liboqs / OpenSSL

The recent Frodo work in Tyr did not change the mathematics. It changed the implementation shape of the public-matrix generator:

  • prefer a persistent OpenSSL EVP_aes_128_ecb context for bulk public-matrix generation,
  • fall back to the local AES-NI bulk path when needed,
  • fall back again to the scalar public fast path when SIMD or OpenSSL is unavailable,
  • generate four rows at a time for \(A S + E\) and eight columns at a time for \(S' A + E'\),
  • use aligned transposed scratch where the AVX2 path wants column-major access.

This is why mulAddAsPlusEStream and mulAddSaPlusEStream look the way they do today. The point was to bring Tyr closer to the layout that liboqs already proved fast, not to create a completely different kernel family.

4. Fast AES table code is allowed only for public data

The scalar helper names are explicit about this: initPublicFast and encryptBlockPublicFast. Their job is public-matrix expansion from seed_A, and seed_A is public by construction. That is why the code accepts a faster public-only AES path here instead of forcing every matrix byte through the secret-handling AES discipline used for private material in other contexts.

In other words: this was not a silent weakening. It was a scoped decision tied to the fact that \(A\) is public-derived.

5. Some things were intentionally not implemented

Not implemented yet

  • the other Round 3 Frodo parameter sets such as frodo640aes and frodo1344aes
  • the SHAKE-based Frodo variants
  • full higher-order masking of the decapsulation path
  • a formal secure-wipe primitive with volatile semantics for all secret buffers

Why not

  • the repo's benchmark and interop target was frodo976aes
  • the fastest alignment work was on the AES public-matrix path, so the SHAKE family would duplicate the main tuning effort
  • masked Frodo is a different engineering problem and needs stronger leakage validation than this repo currently carries
  • plain clear loops are useful hygiene, but moving to a guaranteed optimizer-resistant wipe should be treated as a cross-repo hardening task, not a Frodo-only patch

A second deliberate omission is a decapsulation API that exposes a success bit. The code always returns a shared secret. That keeps the failure condition inside the masked preimage selection logic instead of turning it into an obvious oracle surface.

6. KAT compatibility and liboqs parity were treated as implementation requirements

The Frodo tests do more than "it roundtrips". The KAT path rebuilds the deterministic transcript and checks that Tyr matches the liboqs FrodoKEM-976-AES corpus. That matters because a code path can be fast and still quietly stop being the same construction.

Security, Zeroization, and Side-Channel Notes

1. Constant-time decapsulation compare and selection

The core decapsulation hardening lives in frodo/util.nim and the end of frodo/operations.nim:

  • ctVerifyWords accumulates a word-wise XOR difference over the whole buffer and compresses it to 0 or -1 without early exit.
  • ctSelectBytes copies one candidate secret when the selector is 0 and the fallback secret otherwise, using a byte mask instead of a branch.
  • frodoTyrDecaps always recomputes both expected ciphertext components before selection, instead of returning early on the first mismatch.

In symbols, decapsulation computes a validity mask and then does \[ \text{selected} = \begin{cases} k' & \text{if } (B', C) = (B'', C'') \\ s & \text{otherwise} \end{cases} \] but the implementation realizes that choice with a mask, not a branch.

2. The sampler uses a fixed scan over the CDF table

frodoSampleN does not stop early when it finds the noise bucket. It always walks the same CDF table and accumulates the result with arithmetic. That is the right shape for a secret-bearing sampler:

  • no secret-dependent exit from the loop,
  • no secret-dependent lookup into a wide distribution table,
  • sign handling done with arithmetic masking rather than a branch.

3. Public matrix generation is fast because it is public

The matrix generator is performance-critical, but it is not secret-critical in the same way. The inputs are seed_A, row indices, and column indices, all public. That is why the implementation is comfortable preferring:

  • OpenSSL ECB context reuse,
  • AES-NI bulk encryption,
  • and a scalar public fast AES fallback.

This is a good example of scoping the hardening budget correctly. Constant-time discipline matters most on secret-bearing samplers, comparisons, and preimage selection. It does not need to slow down public-only matrix expansion unnecessarily.

4. Where transient memory is zeroed

Frodo uses clearBytes and clearWords for explicit transient cleanup. The main wipe sites are:

Operation Buffers cleared Why
Key generation seedSEWords, bWords, pkSeedA, pkh, plus the caller randomness copy material These contain secret \(S\), secret noise, seed material, or hashed key state that does not need to live after serialization.
Encapsulation pkh, pkSeedA, g2Out, fin, noiseWords, bpWords, bWords, vWords, cWords, plus the local mu copy These include ephemeral secret noise, candidate secret material, and the final transcript preimage.
Decapsulation muPrime, decSeedA, g2Out, fin, selected, bpWords, cWords, sWords, wWords, noiseWords, bbpWords, bWords, ccWords This is the most sensitive path because it reconstructs both secret state and the masked final preimage.
OpenSSL public AES context Aes128OpenSslCtx is cleared by freeing the EVP_CIPHER_CTX The matrix generator context is public, but it is still good lifecycle hygiene to free it promptly.
Important caveat: clearBytes and clearWords are plain write loops over Nim sequences. That is good practical cleanup, but it is not the same thing as a formally guaranteed optimizer-resistant secure wipe such as a dedicated volatile cleanse primitive.

5. Oracle leaks avoided by design

  • Decapsulation does not expose a success/failure return separate from the shared secret.
  • The compare is whole-buffer and branchless, not "stop at first mismatch".
  • The final preimage selection uses a mask, not an if.
  • The reconstructed ciphertext is always computed before selection, so the attacker cannot distinguish a cheap reject path from a full accept path.

That does not prove full constant-time behavior on every machine. It does remove the obvious source-level decapsulation oracle patterns that tend to appear first: early reject branches, short-circuit compares, and secret-dependent final selection.

Worked Math Examples

Example 1: the matrix multiply shape Frodo actually uses

Ignore real sizes for a moment and work over \(q = 16\). Let

\[ A = \begin{bmatrix} 1 & 2 & 3 & 4 \\ 5 & 6 & 7 & 8 \\ 9 & 1 & 2 & 3 \\ 4 & 5 & 6 & 7 \end{bmatrix}, \quad S = \begin{bmatrix} 1 & 0 \\ 0 & 1 \\ 1 & 1 \\ 0 & 1 \end{bmatrix}, \quad E = \begin{bmatrix} 1 & 0 \\ 0 & 1 \\ 1 & 1 \\ 0 & 0 \end{bmatrix}. \]

Then \[ B = A S + E \pmod{16}. \] The first row of \(A S\) is \[ [1,2,3,4] \begin{bmatrix} 1 & 0 \\ 0 & 1 \\ 1 & 1 \\ 0 & 1 \end{bmatrix} = [4, 9]. \] After adding the first error row \([1, 0]\), the first output row becomes \([5, 9]\).

Tyr's optimized keygen path does not build all of \(A\) at once to do this. It generates a small row block from seed_A, performs exactly this kind of dot-product accumulation for the current block, and then moves on.

Example 2: why the encoded message is added near the top bits

For frodo976aes, the extracted message width is \(B = 3\) bits and \(q = 2^{16}\). Encoding lifts a 3-bit chunk into the high bits of a 16-bit word: \[ \text{encode}(m) = m \cdot 2^{16-3} = m \cdot 2^{13}. \]

If one 3-bit chunk is \(m = 5 = 101_2\), then \[ \text{encode}(5) = 5 \cdot 8192 = 40960. \] Later, decoding rounds the noisy word back down by shifting right by 13 bits after adding the usual midpoint bias. That is why keyEncode and keyDecode only need a few shifts and masks rather than a larger table.

Example 3: masked decapsulation selection

Suppose decapsulation has recomputed candidate key bytes \(k'\) and has fallback secret bytes \(s\) stored in the secret key. Let the selector be:

\[ \text{selector} = \begin{cases} 0 & \text{ciphertext matches} \\ -1 & \text{ciphertext mismatch} \end{cases} \]

Then byte selection is implemented as \[ \text{out}_i = (\neg mask \land k'_i) \lor (mask \land s_i), \] where mask is the selector reinterpreted as a byte mask. If mask = 0x00, the output is \(k'\). If mask = 0xFF, the output is \(s\). No branch is needed.

Papers and Implementation References

The current Tyr Frodo code is best described as Round 3 FrodoKEM-976-AES mathematics plus liboqs-aligned implementation structure. These were the main sources used to shape it:

  • FrodoKEM team, Round 3 specification. This is the direct source for the parameter set, key layout, message encoding, sampling, and keygen / encaps / decaps equations used here.
    FrodoKEM Round 3 specification PDF
  • Joppe Bos et al., "Frodo: Take off the ring! Practical, quantum-secure key exchange from LWE". This is the original design motivation for using standard LWE rather than ring structure. It explains the "large matrix, explicit arithmetic, conservative design" philosophy that still defines the implementation.
    IACR ePrint 2016/659
  • liboqs FrodoKEM implementation. This is the practical implementation reference Tyr was aligned against for performance: streaming row and column kernels, AES-backed matrix generation, and the OpenSSL ECB context shape.
    liboqs FrodoKEM source

The implementation was made in that order:

  1. match the Round 3 FrodoKEM-976-AES algorithm and object layout exactly enough to pass deterministic tests and interoperate with liboqs,
  2. keep the public-matrix generator on-demand instead of materializing the full matrix in the hot path,
  3. align the AES-backed matrix kernels with the liboqs / OpenSSL structure,
  4. keep the decapsulation compare and final selection constant-time,
  5. wipe transient secret buffers at the end of each top-level operation.

It is also important to say what this page is not claiming: the repo does not currently implement the later post-NIST Frodo proposal line, a full masked Frodo, or every parameter set from the broader Frodo family. It documents the code that is actually here now: Round 3 frodo976aes, optimized and hardened within that scope.
