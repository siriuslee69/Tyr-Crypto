High-Level View

Dilithium in Tyr-Crypto

The Tyr Dilithium implementation is a pure-Nim ML-DSA backend organized around four main concerns: parameters, arithmetic, polynomial and vector kernels, and the keypair-sign-verify orchestrator. The optimization story differs from Kyber because sign and verify improved relatively early, while keypair stayed more difficult due to the secret-vector samplers and the matrix-expansion path.

The dominant design tension was the \(\eta\)-sampler. The fastest naive versions had the wrong leakage shape: either the loop count, the SIMD lane completion behavior, or the destination write position depended directly on secret-derived acceptance progress. The accepted work therefore hardened the sampler shape first, then recovered performance in controlled stages.

Sampler-first hardening Secret eta sampling was treated as the first security problem, not an afterthought.
Fixed-width SIMD kernels AVX2 helped most once the vector width matched the fixed Dilithium parameter sets.
Rollback discipline Several theoretically cleaner variants were removed after they lost to the benchmark.

System View

Dilithium has three different hot-path personalities:

  1. Keypair: sample \(s_1\) and \(s_2\), expand matrix rows from \(\rho\), multiply in the NTT domain, and pack key material.
  2. Sign: sample ephemeral state, compute matrix-vector terms, decompose and hint, then hash the transcript.
  3. Verify: unpack, rebuild the challenge polynomial, recompute the matrix-vector term, apply hints, pack \(w_1\), and hash-compare.
Mental model: Dilithium is a pipeline of repeated fixed-shape lattice kernels surrounded by many small post-processing loops. Some of the best gains came from removing repeated dispatch inside those small loops once the big kernels were already in good shape.

That is why the final work was split across two layers: first the large samplers and matrix-vector kernels, then the smaller repeatedly-called helpers such as decompose, useHint, and fixed-width accumulation wrappers.

Hot Path by Operation

Keypair

  1. Derive seeds and expand matrix rows from \(\rho\).
  2. Sample the secret vectors \(s_1\) and \(s_2\) from the eta distribution.
  3. Transform where needed into the NTT domain.
  4. Compute the public matrix-vector product and add error state.
  5. Pack the public and secret keys.

Keypair stayed the hardest operation to close against liboqs because the eta sampler is both a performance hotspot and a side-channel hotspot.

Sign

  1. Expand the matrix and sample ephemeral state.
  2. Compute matrix-vector terms in the NTT domain.
  3. Decompose coefficients, compute hints, and build the challenge relation.
  4. Pack the signature and hash the transcript.

Sign benefited strongly from fixed-width AVX2 accumulation because Dilithium only uses a few specific vector widths, not an arbitrary dynamic width.

Verify

  1. Unpack the public key and signature.
  2. Rebuild the challenge polynomial.
  3. Recompute the relevant matrix-vector term.
  4. Apply hints and repack \(w_1\).
  5. Hash and compare.

Verify improved after moving \(\gamma_2\) dispatch out of the innermost coefficient loops and into once-per-call specialization.

Performance Problems, Solutions, and Rollbacks

1. The original fast eta sampler had the wrong leakage shape

The first fast eta sampler was attractive on paper, but the observable shape was wrong:

  • secret-dependent refill behavior,
  • secret-dependent SIMD lane completion,
  • destination writes driven directly by the running acceptance count.

The accepted fix was not "make it perfectly oblivious at any cost". The accepted fix was:

  1. introduce a fixed-work prefix sampler,
  2. remove the older lane-divergence behavior,
  3. then restore batching only where the end-to-end shape stayed acceptable.

2. The fully oblivious destination-placement fix was correct, but too slow

One attempted design wrote candidates through a masked full-array placement scheme so the final destination would no longer depend on the acceptance count. It was correct and it passed tests. It also hurt keypair badly enough that it had to be rolled back.

Decision rule: a side-channel mitigation that makes the implementation impractical is not the final answer. It is a proof that the shape can be made safer, not proof that this exact version should remain in the tree.

The kept sampler is therefore best described as "substantially harder to observe and much faster", not "fully oblivious regardless of cost".

3. Batching came back in layers

  • AVX2 batching returned for the cleaned eta path.
  • SSE2 x2 leftovers were kept for cases where a full four-lane batch wasted work.
  • Broader x2 leftovers were tried and rolled back when they hurt ML-DSA-87 keypair.

This mattered because the best shape for ML-DSA-65 was not always the best shape for ML-DSA-87. Every generalization had to survive the full benchmark matrix.

4. Fixed-width AVX2 accumulation was one of the clearest wins

Dilithium only uses a few widths for its vector dimensions, especially \(l = 4, 5, 7\). Once the pointwise accumulator was specialized into straight-line AVX2 kernels for those exact widths, the compiler had a much better chance to emit clean vector code.

This improved keypair and sign noticeably, and it did so without complicating the security story.

5. Verify improved when gamma2 dispatch moved out of inner loops

The verify path originally rechecked whether \(\gamma_2 = (q-1)/32\) or \(\gamma_2 = (q-1)/88\) inside coefficient loops. But the variant is fixed for the whole call. The accepted optimization split this into variant-specific helpers and dispatched once at the top.

  • decomposeGamma32 and useHintGamma32
  • decomposeGamma88 and useHintGamma88

The arithmetic stayed the same. The loop control around it got cheaper.

6. Small inline helpers helped only when they matched the generated C well

One real observation from the pass was that Nim's generated C sometimes improves when large compound loops are split into smaller inline helpers. That helped in some places, especially one keypair-focused row-expansion path. It hurt in others, such as an over-specialized verify wrapper that was later removed.

The lesson is precise: inline refactoring is a compiler-shaping tool, not a general law.

7. Strategic wiping was kept, broad wipe-heavy rewrites were not

Secret-bearing exits received secure clears, while public scratch and hot arithmetic buffers kept regular clear paths. This preserved the hardening benefit without turning the implementation into a constant cascade of expensive wipe loops.

8. Summary of the tuning logic

Problem Accepted Solution Why It Stayed
Eta sampler leakage shape Fixed-work prefix sampler before reintroducing batching. It materially improved the side-channel story while preserving speed.
Variable-width accumulation overhead Specialize AVX2 accumulation for \(l = 4, 5, 7\). Dilithium only needs those widths, so specialization paid off cleanly.
Verify inner-loop dispatch Split gamma2 helpers by variant once per call. Same arithmetic, less loop overhead.
Overly clean but slow ideas Roll them back immediately after measurement. The code still has to be practical, not just elegant.
Secret-bearing temporaries Secure clear on strategic exits, regular clear elsewhere. Better hardening without collapsing performance.

Security and Side-Channel Decisions

The Dilithium pass focused on the side channels that are easiest to accidentally introduce while optimizing:

  • secret-dependent loop trip counts,
  • secret-dependent SIMD lane completion,
  • destination writes whose address tracks secret acceptance progress,
  • secret-bearing scratch that survives longer than needed.

The retained sampler still aims at the practical middle ground rather than the strongest conceivable oblivious construction. That is why the fully masked destination-placement variant was rejected even though it was correct: it solved the problem too expensively.

Important distinction: removing secret-dependent branches is not enough on its own. Completion behavior and memory-write shape can also leak secret-dependent progress.

That insight shaped both the sampler changes and the later choice to place secure wiping only at strategic exits instead of everywhere.

Worked Math Examples

Example 1: Eta = 4 nibble sampling

In the \(\eta = 4\) path, each nibble is accepted only if it is smaller than \(9\), and then mapped to \(4 - t\).

Take the byte \(\texttt{0x3A}\). It splits into

\[ t_0 = 10,\qquad t_1 = 3 \]

The first nibble is rejected because \(10 \not< 9\). The second is accepted and becomes

\[ 4 - 3 = 1 \]

This tiny example shows why destination placement matters: one nibble advances the accepted coefficient count while the other does not.

Example 2: Decompose and useHint for \(\gamma_2 = (q-1)/32\)

For ML-DSA-65 and ML-DSA-87,

\[ q = 8380417,\qquad \gamma_2 = \frac{q - 1}{32} = 261888,\qquad 2\gamma_2 = 523776 \]

Choose \(a = 600000\). Then

\[ a_1 = 1,\qquad a_0 = 600000 - 523776 = 76224 \]

If the hint bit is \(1\) and \(a_0 > 0\), then the corrected high part becomes

\[ (a_1 + 1) \bmod 16 = 2 \]

The optimization here was not to change that arithmetic. It was to stop rechecking the \((q-1)/32\) case on every coefficient.

Example 3: Decompose and useHint for \(\gamma_2 = (q-1)/88\)

For ML-DSA-44,

\[ \gamma_2 = \frac{q - 1}{88} = 95232,\qquad 2\gamma_2 = 190464 \]

Choose \(a = 250000\). One valid decomposition is

\[ a_1 = 1,\qquad a_0 = 250000 - 190464 = 59536 \]

If the hint says the high part must move upward, the corrected value is computed in the small \((q-1)/88\) residue class rather than the \((q-1)/32\) one. That is exactly why once-per-call gamma2 specialization is worthwhile.

Example 4: Fixed-width pointwise accumulation

Suppose a toy fixed-width accumulation uses

\[ u = (1, 2, 3, 4),\qquad v = (5, 6, 7, 8) \]

Then the dot product is

\[ 1 \cdot 5 + 2 \cdot 6 + 3 \cdot 7 + 4 \cdot 8 = 5 + 12 + 21 + 32 = 70 \]

The optimization lesson is not the number \(70\). It is that a fixed width lets the compiler emit straighter vector code than a generic loop over a runtime width.
