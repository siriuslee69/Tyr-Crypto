High-Level View

SPHINCS+ in Tyr-Crypto

The Tyr SPHINCS+ backend is a pure-Nim implementation of SPHINCS+-SHAKE-128f-simple. The work followed a strict order: first line the scalar control flow up with the liboqs / PQClean clean implementation, then optimize the shared SHAKE core because SPHINCS+ is hash-dominated, then add SIMD only where the end-to-end benchmark improved without changing the observable safety shape.

The resulting design is intentionally conservative. We kept the algorithm structure close to the standard and to the liboqs reference, used fixed-size arrays instead of heap-heavy sequence traffic on hot paths, and introduced constant-time masking only where secret-driven branch or store patterns had to be removed. We did not fork a separate SPHINCS-only Keccak backend, and we did not adopt more radical signer remappings that would have broken deterministic compatibility with the current verification model.

Reference-first tuning The scalar layout was pulled closer to liboqs / PQClean before deeper Nim-specific work.
Hash-first acceleration The most profitable wins came from fixed-shape SHAKE256 specialization, not from tree logic alone.
Selective hardening Constant-time masking and secure wipes were added where secret-bearing state justified the cost.

System View

At the top level, SPHINCS+ is a stack of three ideas:

  1. hash the message into a FORS digest plus a hypertree position,
  2. sign that digest with FORS to get one root,
  3. authenticate that root upward through a hypertree of WOTS+ signatures and Merkle paths.
Mental model: SPHINCS+ is not "one big tree". It is FORS for message compression plus a hypertree of many tiny WOTS+ / Merkle steps, all driven by a large number of short fixed-shape hashes.

That last point explains most of the implementation strategy. The dominant cost is not complicated branching. The dominant cost is repeatedly hashing buffers of known sizes. This is why the most important optimization pass ended up in the shared SHA3/SHAKE module rather than in a SPHINCS-specific control function.

The signature structure is

\[ \sigma = R \,\|\, \mathrm{SIG}_{\mathrm{FORS}} \,\|\, \prod_{i=0}^{d-1} \left( \mathrm{SIG}^{(i)}_{\mathrm{WOTS}} \,\|\, \mathrm{AUTH}^{(i)} \right) \]

where \(R\) is the message randomizer, \(\mathrm{SIG}_{\mathrm{FORS}}\) signs the message digest portion, and each WOTS+ plus auth-path pair authenticates one root to the next tree layer.

Implemented Variant and Signature Layout

The current local backend implements exactly one parameter set: SPHINCS+-SHAKE-128f-simple. That choice was deliberate: it matched the liboqs comparison target used during the optimization pass, and it avoided overgeneralizing the code before the hot path was understood.

Parameter Value Meaning
\(n\) 16 Byte width of core hash outputs and secret/public seed components.
\(h\) 66 Total hypertree height.
\(d\) 22 Number of subtree layers, so each subtree has height \(h / d = 3\).
\(k\) 33 Number of FORS trees.
\(a\) 6 Height of each FORS tree.
\(w\) 16 WOTS+ Winternitz base.
\(\mathrm{len}\) 35 WOTS+ chain count: 32 message digits plus 3 checksum digits.

The final signature size is

\[ |\sigma| = n + \mathrm{FORS}_{\mathrm{bytes}} + d \cdot \mathrm{WOTS}_{\mathrm{bytes}} + h \cdot n \]

\[ = 16 + 3696 + 22 \cdot 560 + 66 \cdot 16 = 17088 \text{ bytes} \]

In Tyr this parameter record lives in params.nim, and the implementation stays mostly on fixed arrays because these values are compile-time known.

Hot Path by Operation

Key Generation

  1. Take a 48-byte seed and split it into SK.seed, SK.prf, and PK.seed.
  2. Build a signing context from SK.seed and PK.seed.
  3. Generate the top Merkle root.
  4. Assemble SK = SK.seed || SK.prf || PK.seed || root and PK = PK.seed || root.

Key generation is mostly "run the signer support machinery once to compute the public root". That is why several security fixes in the signer path also matter to keygen cleanup.

Signing

  1. Compute the per-message randomizer \(R\) from SK.prf, optrand, and the message.
  2. Hash \(R\), the public key, and the message to get:
    • a FORS digest,
    • a 63-bit tree index,
    • a 3-bit leaf index.
  3. Use FORS to sign the digest and derive one root.
  4. For each of the 22 subtree layers, WOTS-sign the current root and append the auth path.

The hot loops here are WOTS chain generation, auth-path construction, and the many short hash calls inside both. This is why the final sign speed depended strongly on one-block SHAKE fast paths and safe batching for WOTS leaf generation.

Verification

  1. Recompute the same digest, tree index, and leaf index from \((R, PK, M)\).
  2. Rebuild the FORS public key from the signature.
  3. For each subtree layer, rebuild the WOTS public key from the signature, hash it into a leaf, and climb one auth path.
  4. Compare the final reconstructed root against the public key root.

Verification branches only on public inputs. That matters for the security story: it means SIMD scheduling on verify can remain branchy as long as the branch condition depends only on public signature data.

Design Decisions, Optimizations, and Non-Adopted Paths

1. Match liboqs / PQClean first

The scalar Tyr path was first brought closer to the liboqs / PQClean clean implementation: fixed stack buffers, similar address flow, and similar top-level control order. This reduced "Nim noise" before deeper optimization began.

Design rule: if the pure-Nim control flow solves the same subproblem in a different order from the clean reference, performance debugging becomes guesswork. The 1:1 fit stage came first for that reason.

2. Hash specialization mattered more than tree cleverness

The highest-impact optimization was not a new tree traversal. It was a faster path for the repeated short SHAKE256 calls used by thash and prfAddr.

  • The dominant SPHINCS inputs fit in one SHAKE256 rate block.
  • That enabled a fixed-shape absorb-plus-permute fast path in the shared Keccak backend.
  • The SPHINCS wrapper in hash.nim then routes the hot one-block and two-block cases into that path.

We did not create a separate SPHINCS-only SHAKE algorithm. We kept one shared SHA3/SHAKE implementation and specialized the hot entry points instead. That preserves auditability and test coverage across users of the shared hash module.

3. SIMD stayed only where the batch shape was real

AVX2 helped when four lanes truly represented four independent SPHINCS hashes with the same step:

  • prfAddrBatch4 for parallel secret-node generation,
  • thash1Batch4 for one-block node hashing,
  • safe verify-side WOTS batching only when all active lanes are at the same public chain step.

SIMD did not stay enabled everywhere. Early versions that looked attractive in a microbenchmark but lost to scalar end-to-end were rolled back.

4. Nim-specific cleanup mattered

  • Hot buffers were kept on fixed arrays.
  • Sequence churn was avoided on the signer hot path.
  • Casts were kept mostly at SIMD pack boundaries rather than spread through the tree logic.
  • Memory copies were reduced by using caller-owned buffers and exact-size stack temporaries.

This was the right place to use Nim-specific optimization effort. It improved speed without changing the cryptographic structure.

5. Why some ideas were not implemented

Idea Why it was not kept or not adopted
A separate SPHINCS-only Keccak backend The shared SHA3/SHAKE backend already serves the required fast paths, and forking it would duplicate audit burden.
Broader cast-heavy buffer reinterpretation The speed gain was not strong enough to justify hiding layout assumptions in more places.
Full adoption of the SLH-DSA-like side-channel resistant signer from Fluhrer Inference from the paper: it changes the signer mapping and comes with a noticeable slowdown, while the current goal was compatibility with the existing verification and deterministic test flow.
Hardware-style SLH-DSA padding accelerators The idea is valid, but the main paper here targets dedicated hardware rather than the current portable Nim software backend.
Additional SPHINCS variants beyond SHAKE-128f-simple The current pass optimized the exact liboqs comparison target first instead of spreading effort over unoptimized variants.

One platform-specific constraint also mattered: on this Windows machine, the upstream liboqs SPHINCS AVX2 build was not directly available for a true local AVX2-vs-AVX2 comparison. That did not block Tyr's own AVX2 work, but it did limit direct apples-to-apples upstream measurement.

Security, Constant-Time Behavior, and Zeroization

Constant-time and side-channel decisions

SPHINCS+ does not have the same signing-failure oracle problem as rejection-based lattice signatures, because signing does not repeatedly reject secret-dependent candidates until one is accepted. The main side-channel question here is different: do secret-derived values change branches, lane activity, or memory write positions?

The final Tyr hardening addressed the signer-side cases that mattered:

  • WOTS signature element capture in merkle.nim is now masked instead of branching on the selected chain target.
  • Auth-path selection in merkle_utils.nim is now masked instead of conditionally copying the matching sibling.
  • The helper pair ctMaskEqU32 and ctCopyBytesMasked implements those branchless selections.
  • No secret-indexed lookup tables were introduced in the SPHINCS layer or in the shared Keccak backend.
Important nuance: some branches still exist in verification and root-rebuild code, but those branches are driven by public signature or tree-position data, not by hidden signer state.

The final public API also avoids an unnecessary oracle surface: verification returns a final boolean instead of exposing stage-by-stage failure reasons. That is not a constant-time issue by itself, but it avoids turning internal reconstruction state into externally visible protocol behavior.

Memory zeroization

The zeroization policy is intentionally split in two:

  • zeroMem-based clearing for regular POD scratch and intermediate public-ish state.
  • std/volatile.volatileStore-based clearing only for direct secret-bearing state where the optimizer must not elide the wipe.

This split exists because broad volatile wiping on every hot-path buffer was too expensive. The current implementation therefore wipes only the high-value secrets with volatile stores and uses cheaper clearing for the rest.

Where What gets cleared How
context.nim SphincsCtx holding skSeed and pubSeed clearSensitivePlainData using volatileStore
operations.nim temporary seed material copied during key generation clearSensitiveBytes
merkle.nim secret WOTS chain nodes created from prfAddr clearSensitivePlainData
fors.nim skPart, the direct FORS secret leaf material clearSensitivePlainData
multiple modules digest buffers, roots, step arrays, and other plain scratch clearPlainData / clearBytes via zeroMem

Validation-boundary hardening

The signing entrypoint now validates secret-key length before dereferencing it. This matters because a short-key crash is not just a correctness problem; in release builds it can become an out-of-bounds read surface. The current public signer rejects malformed keys at the API boundary instead of faulting inside context initialization.

Worked Examples

Example 1: digest split for SHAKE-128f-simple

For this variant, the message hash output is split into:

\[ \mathrm{dgstBytes} = \mathrm{forsMsgBytes} + \mathrm{treeBytes} + \mathrm{leafBytes} = 25 + 8 + 1 = 34 \]

The first 25 bytes feed FORS, the next 8 bytes encode 63 tree bits, and the last byte contributes the 3 leaf bits used inside one subtree.

Example 2: WOTS+ base-\(w\) digits

With \(w = 16\), every byte becomes two base-16 digits. As a toy example:

\[ \texttt{0xAB 0x3F} \longrightarrow (10, 11, 3, 15) \]

In a small toy checksum over those four digits, \[ c = \sum (15 - d_i) = (5 + 4 + 12 + 0) = 21 \] The real implementation does this over 32 message digits and then appends 3 checksum digits, giving the 35 WOTS+ chain lengths used in this parameter set.

Example 3: auth path inside one 3-level subtree

Each of the 22 hypertree layers has subtree height 3. Suppose the current leaf index inside one subtree is \(5 = 101_2\).

Level 0 sibling: \(4\)
Level 1 sibling: \(3\)
Level 2 sibling: \(0\)

The signer includes those three sibling nodes as the auth path. Verification rebuilds the path by hashing left/right pairs in the order implied by the public index bits.

Example 4: why hash specialization helped

The hot one-block thash call is:

\[ \mathrm{thash}(X) = \mathrm{SHAKE256}(PK.seed \,\|\, ADRS \,\|\, X) \]

For this variant, \[ |PK.seed| + |ADRS| + |X| = 16 + 32 + 16 = 64 \text{ bytes} \] which fits in one SHAKE256 rate block. That is why a fixed one-block absorb path was worth adding: the common SPHINCS case does not need the more general absorb machinery.

The Tyr scalar structure was intentionally fitted closer to the clean liboqs / PQClean flow before more aggressive tuning. That gave a trustworthy baseline for both correctness and performance work.

Optimization and side-channel papers that informed the implementation

  • Multi-Armed SPHINCS+ reinforced the idea that SPHINCS is highly parallelizable, especially around independent chains.
  • Optimization for SPHINCS+ using Intel Secure Hash Algorithm Extensions reinforced that the hash core is the right target for software acceleration.
  • Accelerating SLH-DSA by Two Orders of Magnitude with a Single Hash Unit reinforced the value of padding-format and iterative-hash specialization, even though its main implementation target is hardware.
  • Side Channel Resistant Sphincs+ informed the side-channel review. Inference: it was not adopted directly because it changes the signer mapping and carries a measurable slowdown.
Implementation summary: the final Tyr SPHINCS backend is best described as "spec-faithful and liboqs-aligned first, then hash-specialized, then selectively SIMD-accelerated, then selectively hardened." That order was important. It kept the code auditable while still moving the hot path substantially closer to the reference implementation's shape.
